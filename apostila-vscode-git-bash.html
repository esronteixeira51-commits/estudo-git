Seção 01

Instalando o VS Code

O Visual Studio Code (VS Code) é um editor de código gratuito e open-source da Microsoft. É leve, altamente extensível e o editor mais popular do mundo entre desenvolvedores.

Download e instalação

1
Acesse o site oficial
Vá em code.visualstudio.com e clique em Download. O site detecta automaticamente seu sistema operacional.
2
Windows — instale o .exe
Execute o instalador VSCodeSetup-x64.exe. Durante a instalação, marque as opções "Adicionar ao PATH" e "Abrir com Code" no menu de contexto — isso facilita muito o uso posterior.
3
macOS — arraste para Applications
Abra o .dmg baixado e arraste o ícone do VS Code para a pasta Applications. Depois, abra o VS Code e pressione Cmd+Shift+P → digite "shell command" → clique em "Install 'code' command in PATH".
4
Linux (Ubuntu/Debian) — via terminal
Use o gerenciador de pacotes ou o snap:
terminal — Ubuntu/Debian
# Opção 1: via snap (mais fácil)
$ sudo snap install --classic code

# Opção 2: via repositório oficial Microsoft
$ wget -qO- https://packages.microsoft.com/keys/microsoft.asc | gpg --dearmor > packages.microsoft.gpg
$ sudo install -D -o root -g root -m 644 packages.microsoft.gpg /etc/apt/keyrings/packages.microsoft.gpg
$ sudo apt-get update
$ sudo apt-get install -y code

Verificando a instalação

Após instalar, abra um terminal e teste:

terminal
$ code --version
1.90.0
a3b5b...

# Abre o VS Code na pasta atual
$ code .

# Abre um arquivo específico
$ code index.html
💡

Dica: O comando code . (com ponto) abre o VS Code na pasta onde você está. É o jeito mais rápido de começar a trabalhar em um projeto.

Seção 02

Instalando o Git

O Git precisa estar instalado no sistema antes de usá-lo no VS Code. O editor detecta o Git automaticamente após a instalação.

Windows

1
Baixe o instalador
Acesse git-scm.com/download/win e baixe o instalador para 64-bit.
2
Configurações importantes durante a instalação
• Editor padrão: selecione Visual Studio Code
• Nome do branch inicial: selecione main
• PATH: selecione "Git from the command line and also from 3rd-party software"
• Line endings: "Checkout Windows-style, commit Unix-style"
3
Reinicie o VS Code após instalar
O VS Code precisa ser reaberto para detectar o Git recém-instalado no PATH do sistema.
🪟

Windows: o instalador do Git também instala o Git Bash — um terminal com emulação Unix que você pode usar dentro do VS Code. Recomendamos configurá-lo como terminal padrão (veja Seção 3).

macOS

terminal — macOS
# Opção 1: via Homebrew (recomendado)
$ brew install git

# Opção 2: via Xcode Command Line Tools
$ xcode-select --install

# Verifique a instalação
$ git --version
git version 2.45.0

Linux (Ubuntu/Debian)

terminal — Linux
$ sudo apt update
$ sudo apt install -y git

$ git --version
git version 2.43.0

Configuração inicial obrigatória

Logo após instalar, configure seu nome e e-mail. Esses dados aparecem em todos os seus commits:

terminal — configuração global
# Seu nome (como vai aparecer nos commits)
$ git config --global user.name "Seu Nome Completo"

# Seu e-mail (use o mesmo do GitHub)
$ git config --global user.email "seu@email.com"

# Define o VS Code como editor padrão do Git
$ git config --global core.editor "code --wait"

# Define "main" como nome padrão do branch inicial
$ git config --global init.defaultBranch main

# Confirme tudo que foi configurado
$ git config --global --list
user.name=Seu Nome Completo
user.email=seu@email.com
core.editor=code --wait
init.defaultBranch=main
ℹ️

O arquivo ~/.gitconfig guarda todas essas configurações globais. Você pode editá-lo diretamente com git config --global --edit (vai abrir no VS Code).

Seção 03

Configurando o Bash no VS Code

O Bash (Bourne Again Shell) é o shell mais popular em sistemas Unix. No Windows, o Git instala o Git Bash. Vamos configurar o VS Code para usar o Bash como terminal padrão.

Definindo o Bash como terminal padrão

1
Abra as configurações do VS Code
Pressione Ctrl+Shift+P (ou Cmd+Shift+P no Mac) e digite "Open User Settings JSON". Clique no resultado para abrir o arquivo settings.json.
2
Adicione a configuração do terminal
Cole o trecho correspondente ao seu sistema operacional:

Windows — Git Bash

settings.json — Windows
{
  "terminal.integrated.defaultProfile.windows": "Git Bash",
  "terminal.integrated.profiles.windows": {
    "Git Bash": {
      "path": "C:\\Program Files\\Git\\bin\\bash.exe",
      "args": ["--login", "-i"],
      "icon": "terminal-bash"
    }
  }
}
⚠️

Atenção Windows: se o Git foi instalado em outro local, ajuste o caminho. Para confirmar onde está o bash, abra o Prompt de Comando e execute: where bash

macOS — Bash ou Zsh

settings.json — macOS
{
  // macOS usa Zsh por padrão (desde Catalina). Para usar Bash:
  "terminal.integrated.defaultProfile.osx": "bash",

  // Ou se preferir manter Zsh (recomendado no Mac):
  "terminal.integrated.defaultProfile.osx": "zsh"
}

Linux — Bash

settings.json — Linux
{
  "terminal.integrated.defaultProfile.linux": "bash"
}

Configurando o arquivo .bashrc

O arquivo ~/.bashrc é executado toda vez que você abre um terminal Bash. Use-o para personalizar seu ambiente:

~/.bashrc — configurações recomendadas
# ── Exibe branch do Git no prompt ────────────────────
parse_git_branch() {
  git branch 2>/dev/null | sed -n '/\* /s///p'
}

# Prompt colorido com branch atual
export PS1="\[\033[32m\]\u@\h\[\033[00m\]:\[\033[34m\]\w\[\033[33m\]\$(parse_git_branch)\[\033[00m\]\$ "

# ── Aliases úteis ─────────────────────────────────────
alias ll='ls -la'
alias gs='git status'
alias ga='git add'
alias gc='git commit -m'
alias gp='git push'
alias gl='git log --oneline --graph'
alias ..='cd ..'
alias ...='cd ../..'

# ── Abre VS Code na pasta atual ───────────────────────
alias vsc='code .'

Após editar o arquivo, recarregue sem fechar o terminal:

terminal
$ source ~/.bashrc

# Agora seu prompt mostra a branch atual!
usuario@maquina:~/projetos/meu-app (main)$
💡

Dica: Para editar o .bashrc com o VS Code, use: code ~/.bashrc

Comandos Bash essenciais

Esses são os comandos do dia a dia para navegar e manipular arquivos no terminal:

bash — navegação e arquivos
# ── Navegação ──────────────────────────────────────────────────
$ pwd                          # mostra onde você está
/home/usuario/projetos

$ ls -la                       # lista com detalhes e ocultos
$ cd meu-projeto               # entra na pasta meu-projeto

$ cd -                         # volta para a pasta anterior

$ mkdir -p a/b/c              # cria pastas aninhadas
$ touch index.html             # cria arquivo vazio
$ cp origem.txt destino.txt    # copia arquivo
$ mv antigo.txt novo.txt       # renomeia/move arquivo
$ rm arquivo.txt               # deleta arquivo
$ rm -rf pasta/                # deleta pasta inteira (cuidado!)

# ── Ver conteúdo de arquivos ───────────────────────────────────
$ cat arquivo.txt              # exibe conteúdo inteiro
$ less arquivo.txt             # exibe com paginação (q para sair)
$ head -n 10 arquivo.txt      # primeiras 10 linhas
$ grep "texto" arquivo.txt    # busca texto no arquivo

Seção 06

Extensões Essenciais

As extensões transformam o VS Code num ambiente completo. Para instalar: pressione Ctrl+Shift+X e pesquise pelo nome.

Extensões para Git

🔍 GitLens

Mostra quem escreveu cada linha de código diretamente no editor (git blame inline), histórico visual, comparação de branches. Essencial

📊 Git Graph

Visualização gráfica do histórico de commits e branches. Excelente para entender merges e rebases visualmente. Essencial

🔀 Git History

Navegue no histórico de qualquer arquivo, veja diffs entre versões e restaure versões antigas com um clique.

🤝 GitHub Pull Requests

Cria e revisa Pull Requests do GitHub diretamente no VS Code, sem precisar abrir o navegador.

Extensões de produtividade geral

🎨 Prettier

Formata automaticamente seu código ao salvar. Suporta HTML, CSS, JavaScript, JSON, Markdown e muito mais.

🌈 Indent Rainbow

Coloriza os níveis de indentação. Evita erros de alinhamento e torna o código muito mais legível.

🐍 Python / Pylance

Suporte completo para Python: IntelliSense, debug, formatação e linting. Essencial se você trabalha com Python.

🌐 Live Server

Inicia um servidor local com hot-reload. Clique direito no HTML → "Open with Live Server".

Instalando extensões via terminal

terminal
# Instala extensões diretamente pelo terminal
$ code --install-extension eamodio.gitlens
$ code --install-extension mhutchie.git-graph
$ code --install-extension esbenp.prettier-vscode
$ code --install-extension ritwickdey.liveserver

# Lista extensões instaladas
$ code --list-extensions

Seção 07

Git pelo Painel Source Control

O VS Code tem integração nativa com o Git. O ícone de ramificação na Activity Bar (ou o atalho Ctrl+Shift+G) abre o painel Source Control.

Iniciando um repositório

1
Abra uma pasta de projeto
Use File → Open Folder ou arraste a pasta para o VS Code. Você também pode usar code . no terminal dentro da pasta.
2
Inicialize o repositório
No painel Source Control, clique em "Initialize Repository". Isso executa git init automaticamente. O ícone na Activity Bar vai mostrar quantas mudanças existem.
3
Entendendo as cores dos arquivos
🟢 Verde (U) — Arquivo novo, não rastreado pelo Git
🟡 Amarelo (M) — Arquivo modificado
🔴 Vermelho (D) — Arquivo deletado
🔵 Azul (R) — Arquivo renomeado

Fazendo seu primeiro commit

1
Stage dos arquivos (git add)
No painel Source Control, clique no + ao lado de cada arquivo para adicioná-lo ao staging. Ou clique no + ao lado de "Changes" para adicionar todos de uma vez.
2
Escreva a mensagem do commit
Na caixa de texto no topo do painel (onde está escrito "Message"), escreva uma mensagem descritiva. Ex: "Adiciona página inicial do projeto"
3
Confirme o commit
Clique no botão ✓ Commit (ou pressione Ctrl+Enter). Pronto — o commit foi registrado!

Visualizando o diff no editor

Clique em qualquer arquivo modificado no painel Source Control para ver o diff lado a lado:

index.html ↔ index.html (Working Tree)
  1  <h1>Bem vindo</h1>
  2  -  <p>Conteúdo antigo</p>  2  +  <p>Conteúdo atualizado com melhorias</p>  3  <footer>Rodapé</footer>
💡

Você pode fazer o stage de linhas individuais dentro de um arquivo! No diff, clique no número da linha com o botão direito → "Stage Selected Ranges". Muito útil para commits precisos.

Desfazendo mudanças pelo painel

No painel Source Control, clique com o botão direito em um arquivo para acessar opções como:

↩️ Discard Changes

Desfaz todas as mudanças do arquivo, voltando ao estado do último commit. Equivale a git restore.

➖ Unstage Changes

Remove o arquivo do staging sem desfazer as mudanças. Equivale a git reset HEAD <file>.

Seção 08

Branches e Merge no VS Code

O VS Code tem integração visual completa com branches — você cria, troca, faz merge e resolve conflitos sem sair do editor. A branch atual aparece sempre na Status Bar (canto inferior esquerdo).

Branches locais vs remotos

Antes de trabalhar com branches, é importante entender a diferença entre o que existe no seu PC e o que existe no GitHub.

terminal
$ git branch -a
* main                      ← branch local (onde você está — o * indica)
  feature/login              ← outra branch local
  remotes/origin/main        ← branch que existe no GitHub
  remotes/origin/feature/login ← branch remota
💻 Branch local

Existe só no seu PC. Aparece sem prefixo. Você edita e commita aqui livremente.

☁️ Branch remoto

Existe no GitHub. Prefixo remotes/origin/. Sincronize com git push e git pull.

Criando e trocando de branch — pelo VS Code

1
Clique no nome da branch na Status Bar
No canto inferior esquerdo aparece ⎇ main (ou o nome da branch atual). Clique nele para abrir o menu de branches.
2
Crie uma nova ou selecione existente
Aparece um menu com "Create new branch..." no topo e todas as branches existentes (locais e remotas) abaixo. Para criar, digite o nome e pressione Enter.
3
Ou use o terminal — mais rápido
git switch -c feature/nome cria e entra.
git switch main troca para existente.
💡

Use nomes descritivos com prefixo: feature/login, bugfix/crash-perfil, hotfix/pagamento. Facilita identificar o propósito de cada branch à primeira vista.

Sincronizando branches com o GitHub

terminal — push e pull de branches
# Primeira vez que envia uma branch nova para o GitHub
$ git push -u origin feature/login

# Depois disso, só precisa de:
$ git push     # envia commits da branch atual
$ git pull     # recebe commits da branch atual

# No painel Source Control do VS Code:
# O botão "Sync Changes" faz push + pull automaticamente
# O número ao lado da nuvem mostra commits pendentes

Fazendo merge — pelo painel e pelo terminal

Depois que uma feature está pronta, você integra ela na branch principal.

terminal — merge
# 1. Vá para a branch que vai receber o merge
$ git switch main

# 2. Faça o merge
$ git merge feature/login
Updating 3e887ab..f1c2d3e
Fast-forward
 login.js | 24 ++++++++++++++++++++++++

# 3. Envie o resultado para o GitHub
$ git push

# 4. Delete a branch — trabalho concluído
$ git branch -d feature/login          # deleta local
$ git push origin --delete feature/login  # deleta no GitHub
ℹ️

Pelo VS Code: Command Palette (Ctrl+Shift+P) → "Git: Merge Branch" → escolha a branch. O VS Code faz o merge e já abre os conflitos se houver.

Resolvendo conflitos de merge no VS Code

Quando duas branches editaram o mesmo trecho, o VS Code exibe os conflitos diretamente no editor com botões de resolução:

style.css — conflito de merge
10  body {
    [Accept Current Change] [Accept Incoming] [Accept Both] [Compare]
11  <<<<<<< HEAD (Current)
12    background: #ffffff;
13  =======
14    background: #f5f2ed;
15  >>>>>>> feature/novo-design (Incoming)
16  }
🟢 Accept Current Change

Mantém sua versão (HEAD). Descarta o que está chegando.

🔵 Accept Incoming Change

Aceita a versão que está chegando. Descarta a sua.

⚡ Accept Both Changes

Mantém as duas versões — você edita manualmente o resultado final.

🔍 Compare Changes

Abre visão lado a lado antes de decidir.

1
Resolva todos os conflitos
Clique numa das opções ou edite o arquivo manualmente. Remova os marcadores <<<<<<<, ======= e >>>>>>>.
2
Faça o stage dos arquivos resolvidos
No painel Source Control, clique no + ao lado de cada arquivo resolvido, ou use git add . no terminal.
3
Finalize o merge com um commit
O VS Code já preenche a mensagem automaticamente. Confirme com Ctrl+Enter ou clique em ✓ Commit.

Seção 09

Git pelo Terminal do VS Code

Usar o Git pelo terminal (dentro do VS Code) dá mais controle e é essencial para operações avançadas. O terminal abre já na pasta do projeto.

Fluxo completo de trabalho

Este é o fluxo que você vai usar no dia a dia para qualquer projeto:

terminal integrado — fluxo completo
# 1. Crie ou entre no projeto
$ mkdir meu-projeto && cd meu-projeto

# 2. Inicialize o Git
$ git init
Initialized empty Git repository in /home/usuario/meu-projeto/.git/

# 3. Crie alguns arquivos
$ touch index.html style.css
$ code .   # abre o projeto no VS Code

# 4. Verifique o estado
$ git status
On branch main
Untracked files:
  index.html
  style.css

# 5. Adicione os arquivos ao staging
$ git add .

# 6. Faça o primeiro commit
$ git commit -m "Primeiro commit: estrutura inicial"
[main (root-commit) 3e887ab] Primeiro commit: estrutura inicial
 2 files changed, 0 insertions(+), 0 deletions(-)

Trabalhando com branches no terminal

terminal — branches
# Criar uma nova branch e trocar para ela
$ git switch -c feature/pagina-contato
Switched to a new branch 'feature/pagina-contato'

# Trabalhe normalmente... faça commits...
$ git add contato.html
$ git commit -m "Adiciona página de contato"

# Volta para main e faz o merge
$ git switch main
$ git merge feature/pagina-contato
Updating 3e887ab..f1c2d3e
Fast-forward
 contato.html | 15 +++++++++++++++

# Deleta a branch (já foi mergeada, não precisamos mais)
$ git branch -d feature/pagina-contato

Verificando o histórico com git log

terminal — histórico
# Log resumido e visual
$ git log --oneline --graph --all
* f1c2d3e (HEAD -> main) Adiciona página de contato
* a4b5c6d Adiciona formulário ao index
* 3e887ab Primeiro commit: estrutura inicial

# Ver quem mudou cada linha de um arquivo
$ git blame index.html

# Ver o que mudou num commit específico
$ git show f1c2d3e

Seção 10

Conectando ao GitHub

O GitHub é a plataforma de hospedagem de repositórios mais popular. Vamos conectar seu repositório local ao GitHub.

Autenticação com o VS Code

1
Faça login no GitHub pelo VS Code
Abra a Command Palette (Ctrl+Shift+P), digite "GitHub: Sign In". O VS Code abre o navegador para autenticação OAuth — basta confirmar.
2
Ou configure via HTTPS com token
Acesse github.com → Settings → Developer Settings → Personal Access Tokens → Tokens (classic). Gere um token com permissões de repo. Use esse token como senha quando o Git pedir autenticação.

Publicando um repositório novo no GitHub

1
Crie o repositório no GitHub
Acesse github.com/new, dê um nome ao repositório, deixe-o público ou privado, e clique em "Create repository". Não marque "Initialize with README" se você já tem commits locais.
2
Conecte e envie pelo terminal
O GitHub mostra os comandos exatos. No terminal do VS Code:
terminal — publicando no GitHub
# Adiciona o repositório remoto como "origin"
$ git remote add origin https://github.com/seu-usuario/meu-projeto.git

# Renomeia para "main" caso o branch local seja outro nome
$ git branch -M main

# Primeiro push — configura o tracking (-u)
$ git push -u origin main
Enumerating objects: 5, done.
Counting objects: 100% (5/5), done.
Writing objects: 100% (5/5), done.
To https://github.com/seu-usuario/meu-projeto.git
 * [new branch]      main -> main
Branch 'main' set up to track remote branch 'main' from 'origin'.

# A partir daí, basta usar:
$ git push    # envia commits para o GitHub
$ git pull    # baixa mudanças do GitHub

Publicar direto pelo painel Source Control

Uma forma ainda mais fácil: após o login no GitHub, o painel Source Control mostra um botão "Publish to GitHub" que faz tudo automaticamente.

Clonando um repositório existente

terminal — clonar repositório
# Clona o repositório e já entra na pasta
$ git clone https://github.com/usuario/repo.git
$ cd repo
$ code .   # abre no VS Code

# Ou clone e abra em uma etapa (Command Palette):
# Ctrl+Shift+P → "Git: Clone" → cole a URL → escolha a pasta

Rotina diária com GitHub

terminal — rotina diária
# ── Início do dia ─────────────────────────────────────────
$ git pull                    # baixa o que seu time commitou

# ── Trabalhe na sua feature ───────────────────────────────
$ git switch -c feature/minha-tarefa
# ... edite os arquivos no VS Code ...
$ git add .
$ git commit -m "Implementa funcionalidade X"

# ── Envie para o GitHub ───────────────────────────────────
$ git push -u origin feature/minha-tarefa
# Crie um Pull Request no GitHub para revisão do time

# ── Após aprovação e merge ────────────────────────────────
$ git switch main
$ git pull                    # pega o merge do PR
$ git branch -d feature/minha-tarefa

Dicas finais para o dia a dia

📌 Commit pequenos e frequentes

Prefira muitos commits pequenos e focados a poucos commits gigantes. Fica muito mais fácil entender o histórico e reverter bugs.

🌿 Uma branch por tarefa

Crie uma branch para cada feature ou correção. Nunca trabalhe diretamente na main. Isso protege o código estável.

📝 Use o .gitignore

Nunca commite node_modules/, .env, arquivos de build ou credenciais. Configure o .gitignore antes do primeiro commit.

🔄 Pull antes de push

Sempre faça git pull antes de começar a trabalhar e antes de fazer push. Evita divergências desnecessárias.

👁️ Revise antes de commitar

Use git diff ou clique nos arquivos no painel Source Control para ver exatamente o que vai entrar no commit.

🔖 Use tags para versões

Marque releases importantes com tags: git tag v1.0.0 e git push --tags. Facilita a identificação de versões.

🎯

Seção 12

Iniciando um Repositório e Publicando no GitHub

Este é o fluxo completo que o próprio GitHub sugere quando você cria um repositório novo. Vamos entender cada comando linha por linha.

O fluxo completo

terminal — fluxo completo do GitHub
echo "# estudo-git" >> README.md
git init
git add README.md
git commit -m "first commit"
git branch -M main
git remote add origin https://github.com/esronteixeira51-commits/estudo-git.git
git push -u origin main

O que cada comando faz

1
echo "# estudo-git" >> README.md
Cria o arquivo README.md e escreve o texto # estudo-git dentro dele.

O operador >> significa "adicionar ao final do arquivo". Se o arquivo não existir, ele é criado automaticamente. O # é a sintaxe Markdown para título H1 — no GitHub, o README aparece formatado na página do repositório.
2
git init
Inicializa o Git na pasta atual. Cria uma pasta oculta .git/ onde todo o histórico, configurações e branches ficam armazenados.

Antes do git init → pasta comum, sem versionamento.
Depois do git init → repositório Git, pronto para rastrear mudanças.
3
git add README.md
Coloca o README.md na staging area (área de preparo). O Git ainda não salvou o arquivo no histórico — você está dizendo "quero incluir este arquivo no próximo commit".

Para adicionar todos os arquivos da pasta de uma vez: git add .
4
git commit -m "first commit"
Salva definitivamente o estado atual dos arquivos preparados no histórico do Git. O -m permite escrever a mensagem inline, sem abrir um editor de texto.

Cada commit recebe um ID único (hash), autor, data e a mensagem que você escreveu. É como um ponto de save permanente.
5
git branch -M main
Renomeia a branch atual para main. O -M força a renomeação mesmo se já existir uma branch com esse nome.

Por padrão, versões antigas do Git criavam a branch inicial com o nome master. Hoje o padrão da indústria (e do GitHub) é main. Este comando garante que o nome esteja correto antes do push.
6
git remote add origin <URL>
Conecta seu repositório local ao repositório remoto no GitHub.

• remote → um repositório em outro lugar (neste caso, o GitHub)
• add → adiciona uma nova conexão
• origin → apelido/nome dado a essa conexão (convenção universal, poderia ser qualquer nome)
• <URL> → endereço do repositório no GitHub

Após este comando, origin é um atalho para a URL completa do GitHub.
7
git push -u origin main
Envia seus commits locais para o GitHub.

• push → empurra (envia) os commits para o remoto
• -u (ou --set-upstream) → configura o "rastreamento": vincula sua branch local main à branch remota origin/main
• origin → o apelido do remoto configurado no passo anterior
• main → a branch que será enviada

Depois de usar -u uma vez, nos próximos pushes basta digitar apenas git push.
💡

Verificando o resultado: após o push, confirme que tudo foi enviado corretamente com git remote -v (mostra os remotes configurados) e git log --oneline (mostra os commits locais).

terminal — verificando o resultado
$ git remote -v
origin  https://github.com/esronteixeira51-commits/estudo-git.git (fetch)
origin  https://github.com/esronteixeira51-commits/estudo-git.git (push)

$ git log --oneline
3e887ab (HEAD -> main, origin/main) first commit

# Agora nos próximos commits, o fluxo é simples:
$ git add .
$ git commit -m "minha atualização"
$ git push     # sem precisar de -u origin main novamente

Seção 13

Corrigindo o Remote — git remote remove origin

Errou a URL do repositório remoto? Conectou no repositório errado? O comando git remote remove origin resolve isso. Entenda quando e como usar.

O que o comando faz

terminal
git remote remove origin

Este comando remove a conexão entre o seu repositório local e o remoto chamado origin. Ele não apaga nenhum arquivo local e não afeta o repositório no GitHub — apenas desfaz o vínculo configurado localmente.

ℹ️

Pense no origin como um contato salvo no celular. O git remote remove origin deleta esse contato da sua agenda — o número (repositório no GitHub) continua existindo, você só não tem mais o atalho para ele.

Quando usar

❌ Digitou a URL errada

Erro de digitação na URL do git remote add origin. O push vai falhar ou ir para o lugar errado.

🔄 Quer trocar de repositório

O projeto vai ser hospedado em outro repositório GitHub, GitLab ou qualquer outra plataforma.

📋 Repositório copiado/clonado

Você clonou um projeto de outra pessoa e quer publicar como seu próprio repositório, não fazer push para o original.

🏢 Mudou de organização

O repositório foi movido para outra organização ou usuário no GitHub e a URL mudou.

Fluxo de correção passo a passo

terminal — corrigindo o remote
# 1. Veja qual remote está configurado atualmente
$ git remote -v
origin  https://github.com/esron/repositorio-errado.git (fetch)
origin  https://github.com/esron/repositorio-errado.git (push)

# 2. Remove a conexão errada
$ git remote remove origin

# 3. Confirme que foi removido (não deve retornar nada)
$ git remote -v
(nenhuma saída — não há remotes configurados)

# 4. Adicione o remote correto
$ git remote add origin https://github.com/esronteixeira51-commits/estudo-git.git

# 5. Confirme a nova URL
$ git remote -v
origin  https://github.com/esronteixeira51-commits/estudo-git.git (fetch)
origin  https://github.com/esronteixeira51-commits/estudo-git.git (push)

# 6. Envie normalmente
$ git push -u origin main

Alternativa mais rápida — trocar a URL sem remover

Se você só quer corrigir a URL sem remover e readicionar o remote, use git remote set-url:

terminal — trocar URL diretamente
# Troca a URL do origin sem precisar remover e readicionar
$ git remote set-url origin https://github.com/esronteixeira51-commits/estudo-git.git

# Confirme a mudança
$ git remote -v
origin  https://github.com/esronteixeira51-commits/estudo-git.git (fetch)
origin  https://github.com/esronteixeira51-commits/estudo-git.git (push)
💡

Dica: git remote set-url é mais rápido para corrigir só a URL. Já o git remote remove é útil quando você quer desconectar completamente o projeto de qualquer remoto, como ao começar um fork ou separar um projeto.

Resumo dos comandos de remote

ComandoO que faz
git remote -vLista todos os remotes configurados e suas URLs
git remote add origin <url>Adiciona um novo remote com o apelido "origin"
git remote remove originRemove completamente o remote "origin"
git remote set-url origin <url>Troca a URL do remote "origin" sem remover
git remote rename origin upstreamRenomeia o remote de "origin" para "upstream"
⚠️

Atenção: o nome origin é apenas uma convenção — o Git aceita qualquer nome. Mas seguir essa convenção é importante para trabalhar em equipe, pois todos esperam que o remote principal se chame origin.

Seção 14

Unificando branches main e master

Este é um problema muito comum: você criou um repositório localmente com git init (que gerou a branch master) e o repositório no GitHub foi criado com a branch padrão main. Agora você tem dois branches com históricos completamente separados e precisa unificá-los.

Entendendo o problema

terminal — diagnóstico da situação
# Primeiro veja o que você tem
$ git branch -a
* master                    ← você está aqui (local)
  remotes/origin/main       ← existe no GitHub com commits diferentes
  remotes/origin/master     ← existe no GitHub também

# Veja o histórico de cada um
$ git log --oneline master
f1c2d3e primeiro commit local
3e887ab estrutura inicial

$ git log --oneline origin/main
a4b5c6d Initial commit (criado pelo GitHub)
7d8e9f0 Add README
⚠️

Os dois branches têm commits que o outro não tem — históricos completamente separados. O Git vai recusar o merge por padrão. É exatamente por isso que precisamos do --allow-unrelated-histories.

A solução — passo a passo

1
Certifique-se de estar no branch master
Vamos receber os commits do main dentro do master, então precisamos estar nele.
2
Fazer o merge permitindo históricos diferentes
Usamos a flag --allow-unrelated-histories para forçar o Git a aceitar o merge mesmo sem histórico em comum.
3
Confirmar o commit de merge
O Git vai abrir o editor de texto para confirmar a mensagem. Salve e feche o editor — ou use -m para passar a mensagem diretamente.
4
Enviar o master unificado para o GitHub
Agora o master local tem todo o histórico. Enviamos para o GitHub.
5
Deletar o branch main remoto
Agora que o master tem tudo, o main remoto não é mais necessário. Removemos ele do GitHub.
terminal — fluxo completo de unificação
# Passo 1 — confirme que está no master
$ git branch
* master

# Passo 2 — merge permitindo históricos não relacionados
$ git merge main --allow-unrelated-histories
Merge made by the 'ort' strategy.
 README.md | 1 +
 1 file changed, 1 insertion(+)

# Passo 3 — confirme o commit de merge (se necessário)
$ git commit -m "Merge branch 'main' into master"

# Passo 4 — envie o master unificado para o GitHub
$ git push origin master
To https://github.com/esronteixeira51-commits/estudo-git.git
   3e887ab..f1c2d3e  master -> master

# Passo 5 — delete o branch main do GitHub
$ git push origin --delete main
To https://github.com/esronteixeira51-commits/estudo-git.git
 - [deleted]         main

# Verifique o resultado final
$ git branch -a
* master
  remotes/origin/master

# Apenas um branch, com todo o histórico unificado ✅

Resumo dos comandos usados nesta operação

ComandoPara que serve aqui
git branch -aVer todos os branches locais e remotos antes de agir
git merge main --allow-unrelated-historiesMesclar mesmo sem histórico em comum entre os branches
git commit -m "..."Confirmar o commit de merge quando o Git pausa e espera
git push origin masterEnviar o master unificado para o GitHub
git push origin --delete mainRemover o branch main do GitHub após a unificação

Seção 15

Exercícios Práticos

Estes exercícios simulam situações reais do dia a dia. Execute cada um no terminal do VS Code. Ao final de cada exercício você saberá exatamente o que aconteceu e por quê.

ℹ️

Antes de começar: crie uma pasta chamada treino-git em algum lugar do seu computador e abra ela no VS Code com code .. Todos os exercícios serão feitos dentro dessa pasta.

Exercício 1 — Seu primeiro repositório completo

Situação: você acabou de criar um projeto do zero e quer versioná-lo no Git.

Objetivo: inicializar o Git, criar arquivos, fazer o primeiro commit e ver o histórico.

exercício 1 — execute no terminal
# 1. Inicialize o repositório
$ git init

# 2. Crie três arquivos
$ echo "# Meu Projeto" >> README.md
$ touch index.html style.css

# 3. Veja o que o Git enxerga
$ git status

# 4. Adicione todos os arquivos
$ git add .

# 5. Faça o primeiro commit
$ git commit -m "first commit: estrutura inicial"

# 6. Veja o histórico
$ git log --oneline
✅

Resultado esperado: o git log deve mostrar 1 commit com a mensagem "first commit: estrutura inicial". Se aparecer, você concluiu o exercício com sucesso.

Exercício 2 — Trabalhando com branches

Situação: você precisa adicionar uma nova funcionalidade sem afetar o código principal.

Objetivo: criar uma branch, fazer commits nela e mesclar de volta ao main.

exercício 2 — execute no terminal
# 1. Crie e entre na nova branch
$ git switch -c feature/pagina-sobre

# 2. Crie um arquivo nessa branch
$ echo "Sobre nós" >> sobre.html
$ git add sobre.html
$ git commit -m "Adiciona página sobre"

# 3. Volte para a branch principal
$ git switch main

# 4. Confirme que sobre.html NÃO existe aqui
$ ls

# 5. Faça o merge da feature
$ git merge feature/pagina-sobre

# 6. Confirme que sobre.html AGORA existe
$ ls

# 7. Delete a branch (trabalho concluído)
$ git branch -d feature/pagina-sobre

# 8. Veja o histórico com o grafo
$ git log --oneline --graph
✅

Resultado esperado: o passo 4 mostra que sobre.html não existe na branch main. Após o merge no passo 6, o arquivo aparece. Isso demonstra que as branches são isoladas entre si.

Exercício 3 — Desfazendo um erro

Situação: você fez um commit com um erro e precisa desfazer sem perder o trabalho.

Objetivo: praticar o git reset e o git restore.

exercício 3 — execute no terminal
# 1. Faça uma mudança e commite (simula um erro)
$ echo "linha com erro" >> README.md
$ git add README.md
$ git commit -m "commit com erro"

# 2. Veja que o commit existe
$ git log --oneline

# 3. Desfaça o commit mas MANTENHA as alterações nos arquivos
$ git reset HEAD^

# 4. Veja que o commit sumiu mas o arquivo ainda tem a linha
$ git log --oneline
$ cat README.md

# 5. Agora descarte a mudança do arquivo também
$ git restore README.md

# 6. Confirme que a linha sumiu do arquivo
$ cat README.md
✅

Resultado esperado: após o git reset HEAD^ o commit some do log mas o arquivo ainda tem a mudança. Após o git restore o arquivo volta ao estado anterior. Você aprendeu a diferença entre desfazer um commit e desfazer uma mudança no arquivo.

Exercício 4 — Simulando o problema main vs master

Situação: você tem dois branches com históricos separados e precisa unificar — exatamente o que acontece quando o GitHub cria um branch diferente do seu local.

Objetivo: praticar o --allow-unrelated-histories em um ambiente controlado.

exercício 4 — execute no terminal
# 1. Crie uma pasta nova para este exercício
$ cd ..
$ mkdir exercicio-merge && cd exercicio-merge

# 2. Crie o primeiro repositório (simula o local)
$ git init
$ echo "projeto local" >> local.txt
$ git add .
$ git commit -m "commit do repositório local"

# 3. Crie um branch "main" separado sem herdar o histórico
$ git checkout --orphan main
$ git rm -rf .
$ echo "projeto github" >> github.txt
$ git add .
$ git commit -m "commit do GitHub (histórico separado)"

# 4. Veja que os dois branches não compartilham histórico
$ git log --oneline --all

# 5. Tente fazer merge normal — vai falhar
$ git switch master
$ git merge main
fatal: refusing to merge unrelated histories

# 6. Use a flag correta para resolver
$ git merge main --allow-unrelated-histories

# 7. Confirme que os dois arquivos agora existem juntos
$ ls
github.txt  local.txt

# 8. Veja o histórico unificado
$ git log --oneline --graph
✅

Resultado esperado: o passo 5 mostra o erro refusing to merge unrelated histories. O passo 6 resolve com a flag. Ao final, os dois arquivos existem juntos e o git log --graph mostra as duas linhas convergindo em um ponto de merge.

Exercício 5 — Publicando no GitHub do zero

Situação: você tem um projeto local pronto e quer publicar no GitHub pela primeira vez.

Objetivo: praticar o fluxo completo de publicação — do git init ao git push.

⚠️

Para este exercício você precisa de uma conta no GitHub e um repositório vazio criado em github.com/new. Crie o repositório sem marcar "Initialize with README".

exercício 5 — fluxo completo de publicação
# 1. Crie e entre na pasta do projeto
$ mkdir meu-primeiro-repo && cd meu-primeiro-repo

# 2. Inicialize e crie o README
$ echo "# Meu Primeiro Repositório" >> README.md
$ git init
$ git add README.md
$ git commit -m "first commit"

# 3. Garanta que o branch se chama main
$ git branch -M main

# 4. Conecte ao repositório do GitHub (substitua pela sua URL)
$ git remote add origin https://github.com/SEU-USUARIO/SEU-REPOSITORIO.git

# 5. Confirme a URL antes de enviar
$ git remote -v

# 6. Envie para o GitHub
$ git push -u origin main

# 7. Acesse o GitHub e confirme que o README aparece na página do repositório ✅
✅

Resultado esperado: ao abrir a URL do repositório no navegador você verá o arquivo README.md renderizado com o título "Meu Primeiro Repositório". A partir de agora, para enviar novos commits basta usar git push.
