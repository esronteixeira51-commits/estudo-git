1. Introdução à Fase 2

Na Fase 1 você aprendeu a trabalhar sozinho. Agora vamos aprender a trabalhar em equipe, que é onde o Git brilha.

Branches permitem que várias pessoas (ou você mesmo) trabalhem em funcionalidades diferentes ao mesmo tempo, sem interferir no código principal.

2. O que são Branches?

Uma branch (ramo) é uma linha independente de desenvolvimento. O branch padrão hoje é chamado main.

Analogia: Imagine que o branch main é a estrada principal. Cada nova funcionalidade é uma rua paralela (branch). Quando a rua está pronta, você "mescla" ela de volta na estrada principal.

Tipos de branches mais comuns:

  • main → Branch principal (produção)
  • feature/nome-da-feature → Nova funcionalidade
  • bugfix/nome-do-bug → Correção de erro
  • hotfix/urgente → Correção crítica em produção

3. Comandos Essenciais de Branches

Criar e mudar de branch

git branch                  # lista todas as branches
git branch feature/login     # cria uma branch
git checkout feature/login   # muda para a branch
# Ou tudo em um único comando (mais usado):
git checkout -b feature/cadastro

Ver em qual branch você está

git branch
git status

Renomear branch

git branch -m nome-antigo nome-novo

Deletar branch

git branch -d feature/login     # local
git push origin --delete feature/login  # remota

4. Merge e Resolução de Conflitos

Fazer merge

git checkout main
git pull origin main
git merge feature/login
Conflitos de Merge
Quando duas pessoas editam a mesma linha do mesmo arquivo, o Git não sabe qual versão manter. Você precisa resolver manualmente.

Como resolver conflitos:

  1. O Git marca as partes conflitantes com <<<<<<< HEAD, ======= e >>>>>>> feature/xxx
  2. Edite o arquivo e escolha ou combine o código correto
  3. Remova os marcadores
  4. git add nome-do-arquivo
  5. git commit (sem mensagem - o Git já coloca uma padrão)

5. GitHub Flow (Fluxo Recomendado)

Fluxo simples e moderno, muito usado atualmente:

  1. Crie uma branch a partir da main
  2. Faça commits na sua branch
  3. Abra um Pull Request
  4. Faça Code Review
  5. Quando aprovado → Merge no GitHub
  6. Delete a branch
Vantagem: A branch main fica sempre estável e pronta para deploy.

6. Pull Requests (PRs) no GitHub

Pull Request é a forma mais profissional de contribuir código.

Passo a passo no GitHub:

  1. Faça push da sua branch: git push -u origin feature/nome
  2. Vá no GitHub → seu repositório → clique em "Compare & pull request"
  3. Escreva uma boa descrição
  4. Adicione reviewers (se tiver)
  5. Espere aprovação e faça merge
Dicas para um bom Pull Request:
  • Título claro e objetivo
  • Descreva o que foi feito e por quê
  • Adicione prints se for interface
  • Linke issues relacionadas

7. Fork + Pull Request (Contribuição em Open Source)

Para contribuir em projetos de outras pessoas:

  1. Clique em "Fork" no repositório original
  2. Clone o seu fork: git clone https://github.com/SEU-USUARIO/repo.git
  3. Crie uma branch: git checkout -b feature/minha-contribuicao
  4. Faça as alterações e commit
  5. Push para o seu fork
  6. Abra um Pull Request do seu fork para o repositório original

8. Git Stash (Guardar mudanças temporariamente)

Útil quando você precisa trocar de branch sem commitar mudanças incompletas.

git stash                # guarda as mudanças
git stash list           # lista os stashes
git stash pop            # aplica o último stash e remove da lista
git stash apply          # aplica sem remover da lista
git stash drop           # remove um stash

9. Boas Práticas da Fase 2

  • Mantenha a branch main sempre limpa e estável
  • Nomeie branches de forma clara: feature/, bugfix/, hotfix/
  • Faça commits pequenos e frequentes
  • Sempre atualize sua branch antes de criar um PR: git pull origin main
  • Delete branches após o merge
  • Use repositórios públicos para praticar contribuições

10. Exercícios Práticos (Faça todos!)

  1. Crie um repositório no GitHub chamado aprendendo-git-fase2
  2. Clone o repositório e crie uma branch feature/readme
  3. Melhore o README.md com seções, badges e emojis
  4. Faça commit e push da branch
  5. Abra um Pull Request no GitHub e faça o merge você mesmo
  6. Crie uma branch feature/contato, adicione um arquivo e simule um conflito (edite o mesmo arquivo na main)
  7. Resolva o conflito e faça o merge
  8. Fork um repositório open-source com label "good first issue" e tente fazer uma contribuição pequena
