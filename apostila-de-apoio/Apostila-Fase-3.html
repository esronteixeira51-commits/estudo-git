2. Desfazendo Mudanças (Undo Operations)

git reset

ModoComandoO que acontece
--softgit reset --soft HEAD~1Desfaz o commit, mas mantém as alterações no Staging
--mixed (padrão)git reset HEAD~1Desfaz o commit e tira do Staging
--hardgit reset --hard HEAD~1Desfaz tudo (cuidado! perde alterações)

git revert

git revert HEAD          # cria um commit que desfaz o último commit
git revert abc1234       # desfaz um commit específico
Atenção:
git reset --hard é perigoso. Use apenas se tiver certeza. Prefira git revert em branches compartilhadas.

3. Interactive Rebase (O comando mais poderoso)

Rebase permite "reorganizar" o histórico de commits de forma limpa.

git rebase -i HEAD~5     # interativo nos últimos 5 commits

No editor que abre, você pode:

  • pick → manter o commit
  • reword → mudar a mensagem
  • edit → editar o commit
  • squash / fixup → juntar commits
  • drop → remover o commit
Regra de ouro: Nunca faça rebase em branches que já foram compartilhadas com outras pessoas (principalmente a main).

4. Amend e Squash de Commits

Alterar o último commit

git commit --amend               # muda mensagem ou adiciona arquivos
git commit --amend --no-edit    # só adiciona arquivos sem mudar mensagem

Squash (juntar commits)

Muito usado antes de abrir Pull Request para deixar o histórico limpo.

5. Reflog - O maior salva-vidas do Git

O reflog registra tudo que você fez no repositório, mesmo commits "perdidos".

git reflog
git reflog --oneline

Recuperar um commit perdido:

git reset --hard abc1234   # usando o hash do reflog
Dica profissional: Sempre que achar que "perdeu" algo no Git, rode git reflog primeiro.

6. Cherry-pick

Permite copiar um commit específico de uma branch para outra.

git checkout main
git cherry-pick abc1234     # copia o commit com esse hash

7. GitHub Avançado

Recursos importantes:

  • Branch Protection Rules → Protege a branch main (exige PR aprovado, status checks, etc.)
  • CODEOWNERS → Define quem deve revisar cada pasta/arquivo
  • GitHub Projects → Kanban integrado
  • Discussions → Fórum dentro do repositório
  • Releases e Tags → Versionamento semântico (v1.2.3)
  • GitHub Pages → Hospedar site estático gratuitamente

Como criar um CODEOWNERS

Crie o arquivo .github/CODEOWNERS na raiz:

# Exemplo
*                   @seuusuario
/src/frontend/      @time-frontend
/docs/              @time-docs

8. GitHub CLI (gh) - Trabalhar sem sair do terminal

Instalação: cli.github.com

gh auth login
gh repo create
gh pr create
gh pr list
gh issue create
Vantagem: Você consegue criar PRs, issues e revisar código direto do terminal, muito mais rápido.

9. Boas Práticas Profissionais

  • Use commits atômicos (uma mudança por commit)
  • Escreva mensagens claras seguindo Conventional Commits
  • Mantenha o histórico limpo (rebase + squash antes do merge)
  • Nunca faça git push --force em branches compartilhadas (use --force-with-lease)
  • Faça review de código com atenção
  • Configure proteção de branch na main

10. Exercícios Práticos (Essenciais para virar profissional)

  1. Crie um repositório e faça vários commits. Depois use git reset --soft, --mixed e --hard para entender a diferença.
  2. Faça um Interactive Rebase e pratique reword, squash e drop.
  3. Simule perder um commit e recupere usando git reflog.
  4. Configure Branch Protection Rules na branch main do seu repositório.
  5. Crie um arquivo .github/CODEOWNERS.
  6. Instale o GitHub CLI e crie um Pull Request usando apenas o terminal.
  7. Limpe o histórico de um repositório de testes usando rebase interativo.
