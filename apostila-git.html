01

Manipulação de Arquivos e Pastas

Tudo que envolve criar, mover, renomear e deletar arquivos — tanto no sistema operacional quanto no controle do Git.

Repositório — criar e clonar

iniciar e clonar
$ git init                              # cria repositório na pasta atual
$ git clone <url>                       # clona repositório remoto
$ git clone <url> outro-nome           # clona com nome de pasta diferente

Mover, renomear e deletar

git mv e git rm
$ git mv antigo.txt novo.txt           # renomeia e já stagea
$ git mv arquivo.txt pasta/            # move para outra pasta e já stagea
$ git rm arquivo.txt                   # deleta arquivo e stagea remoção
$ git rm --cached segredo.txt         # remove do Git mas mantém no disco
$ git clean -fd                        # remove arquivos não rastreados (-f força, -d inclui pastas)
⚠️ git rm --cached é essencial quando você commitou acidentalmente um arquivo que deveria estar no .gitignore (como um .env). Ele remove do Git sem deletar do disco.

.gitignore — ignorando arquivos

O .gitignore diz ao Git quais arquivos nunca devem ser rastreados. Crie na raiz do projeto antes do primeiro commit.

exemplo de .gitignore
# Dependências
node_modules/
venv/
__pycache__/

# Builds
dist/
build/
*.pyc

# Variáveis de ambiente — NUNCA commitar
.env
.env.local
*.log

# Sistema operacional
.DS_Store
Thumbs.db
💡 Use gitignore.io para gerar automaticamente um .gitignore para sua stack. Se você já commitou um arquivo que deveria ignorar: git rm --cached arquivo.txt → adicione ao .gitignore → commit.

Restaurar um arquivo de uma versão antiga

restaurar arquivo específico
$ git restore <arquivo> --source HEAD~3   # versão de 3 commits atrás
$ git restore <arquivo> --source v1.0.0   # versão de uma tag específica
$ git checkout <commit> -- <arquivo>       # forma alternativa (mais antiga)
02

Staging e Status

O staging é a área intermediária onde você prepara exatamente o que vai entrar no próximo commit.

fluxo de trabalho
  Working Dir  →  Staging Area  →  Repositório
   (editar)         (git add)         (git commit)

git add — preparar arquivos

git add
$ git add index.html          # adiciona arquivo específico
$ git add src/               # adiciona toda uma pasta
$ git add .                  # adiciona tudo
$ git add -p                # modo interativo — escolhe partes de cada arquivo

git status — verificar estado

git status
$ git status                 # estado completo
$ git status -s             # versão resumida
On branch main
Changes to be committed:       ← staged (vai no próximo commit)
  modified: index.html

Changes not staged for commit: ← modificado mas não staged
  modified: style.css

Untracked files:               ← novo, Git ainda não rastreia
  novo.js

Remover do staging (unstage)

unstage
$ git reset <arquivo>        # remove arquivo do staging (mantém mudanças)
$ git reset                  # remove tudo do staging
$ git restore --staged <arquivo>  # forma moderna de unstage
03

Fazendo Commits

Um commit é um snapshot permanente do projeto. Cada commit tem ID único, autor, data e mensagem.

Criar commits

git commit
$ git commit                          # abre editor para escrever mensagem
$ git commit -m 'mensagem'           # mensagem inline
$ git commit -am 'mensagem'          # stagea arquivos rastreados + commita
$ git commit --amend                 # corrige o último commit (mensagem ou arquivo)
$ git commit --amend -m 'nova msg'  # corrige só a mensagem
🚨 Nunca use --amend em commits já enviados ao repositório remoto. Reescreve o histórico e causa conflitos para quem está colaborando.

Boas mensagens de commit

✅ Boas mensagens

• "Adiciona validação de email no cadastro"
• "Corrige crash ao abrir perfil vazio"
• "Refatora módulo de autenticação"

❌ Mensagens ruins

• "fix"
• "alterações"
• "WIP"
• "aaaaa"

04

Inspecionando o Histórico

Ferramentas para navegar, investigar e entender o histórico de commits.

git log — navegar no histórico

git log
$ git log                        # histórico completo
$ git log --oneline              # uma linha por commit
$ git log --oneline --graph     # árvore visual de branches
$ git log --oneline --graph --all  # inclui todos os branches
$ git log <arquivo>              # commits que tocaram esse arquivo
$ git log --follow <arquivo>    # mesmo após renomear o arquivo
$ git log -G "texto"            # commits que adicionaram/removeram esse texto
$ git log --author="Nome"       # commits de um autor específico
exemplo — git log --oneline --graph --all
* f1c2d3e (HEAD -> main) Corrige validação de email
* 3e887ab Adiciona tela de login
| * a4b5c6d (feature/dashboard) Cria gráfico
| * 7d8e9f0 Estrutura do dashboard
|/
* 1a2b3c4 Configuração inicial

git diff — visualizar diferenças

git diff
$ git diff                    # mudanças não staged
$ git diff --staged           # mudanças staged vs último commit
$ git diff HEAD               # todas as mudanças (staged + unstaged)
$ git diff <c1> <c2>         # diferença entre dois commits
$ git diff <commit> <arquivo> # mudanças em um arquivo desde um commit
$ git diff <commit> --stat   # resumo de quantas linhas mudaram
$ git show <commit>           # conteúdo + diff de um commit
$ git show <commit> --stat   # resumo de um commit

git blame — quem escreveu cada linha

git blame
$ git blame <arquivo>
3e887ab (Ana Silva   2024-03-10) def validar_email(email):
f1c2d3e (João Costa  2024-03-12)     if "@" not in email:
f1c2d3e (João Costa  2024-03-12)         return False
💡 No VS Code, a extensão GitLens mostra o git blame inline enquanto você edita — sem precisar rodar o comando.
05

Desfazendo e Corrigindo

O Git oferece várias formas de reverter mudanças. A escolha depende de onde a mudança está e se já foi publicada.

Descartar mudanças não commitadas

restore e reset
$ git restore <arquivo>                  # descarta mudanças unstaged de um arquivo
$ git restore --staged --worktree <arquivo>  # descarta staged + unstaged de um arquivo
$ git reset --hard                      # descarta TUDO (staged + unstaged) ⚠️
$ git clean -fd                         # remove arquivos não rastreados

Desfazer commits locais (não publicados)

git reset
$ git reset HEAD^                # desfaz último commit — mantém arquivos editados
$ git reset HEAD~3               # desfaz os últimos 3 commits — mantém arquivos
$ git reset --hard HEAD^        # desfaz commit E descarta mudanças ⚠️

Reverter commits já publicados no remoto

Quando o commit já foi publicado, não use reset — ele reescreve o histórico. Use git revert, que cria um novo commit desfazendo as mudanças sem apagar o histórico.

git revert — seguro para commits publicados
$ git revert HEAD               # cria commit que desfaz o último commit
$ git revert <commit-id>        # cria commit que desfaz um commit específico
$ git revert HEAD --no-commit  # desfaz mas não commita (você revisa antes)

# Exemplo:
# Histórico antes:  A ← B ← C (commit com erro)
# Após git revert:  A ← B ← C ← C' (novo commit que desfaz C)
# O histórico fica intacto — o GitHub não rejeita o push
git reset

Reescreve o histórico. Use apenas para commits locais que ainda não foram publicados.

git revert

Cria um novo commit desfazendo. Seguro para commits já no remoto — o histórico fica intacto.

Stash — guardar trabalho temporariamente

git stash
$ git stash                    # guarda mudanças atuais
$ git stash push -m "nome"    # guarda com descrição
$ git stash list               # lista todos os stashes
$ git stash pop                # recupera o último stash e remove da lista
$ git stash apply              # recupera sem remover da lista
$ git stash drop               # descarta o último stash

Desfazer rebase que deu errado

git reflog — recuperação de emergência
$ git reflog                   # histórico de todos os HEADs anteriores
3e887ab HEAD@{0}: rebase (finish): ...
f1c2d3e HEAD@{1}: commit: Tela de login  ← este é o ponto que quero

$ git reset --hard f1c2d3e    # volta para antes do rebase

Descartar mudança em arquivogit restore <arquivo>Médio
Tirar arquivo do staginggit reset <arquivo>Baixo

07

Gerenciando Branches

Branches são linhas independentes de desenvolvimento. Isole funcionalidades novas sem afetar o código estável.

Criar, trocar e listar

git branch e git switch
$ git branch                          # lista branches locais
$ git branch -a                        # lista locais + remotos
$ git branch --sort=-committerdate    # lista por commit mais recente
$ git switch <nome>                   # troca para branch existente
$ git switch -c <nome>               # cria e entra na branch
$ git switch -                        # volta para a branch anterior

Deletar branches

deletar local e remoto
$ git branch -d <nome>              # deleta branch local (só se já foi mergeada)
$ git branch -D <nome>              # força deleção local (mesmo sem merge)
$ git push origin --delete <nome>  # deleta branch no GitHub/remoto
ℹ️ git push origin --delete <nome> remove o branch do servidor remoto. Não afeta nada no seu repositório local. Use após o merge da branch ter sido aprovado.

Renomear branch

renomear
$ git branch -m novo-nome          # renomeia a branch atual
$ git branch -M main               # renomeia para main (força mesmo se existir)
💡 Use nomes descritivos: feature/login, bugfix/crash-perfil, hotfix/pagamento. Facilita muito identificar o propósito de cada branch.
08

Mesclando Branches

Existem quatro formas principais de integrar o trabalho de uma branch em outra.

Merge — preserva histórico completo

git merge
$ git switch main
$ git merge feature/login              # merge com commit de merge
$ git merge --allow-unrelated-histories main  # merge de branches sem ancestral comum

Rebase — histórico linear

git rebase
$ git switch feature/login
$ git rebase main                      # reaplica commits sobre a ponta da main
$ git rebase -i HEAD~5                # rebase interativo — edita últimos 5 commits
🚨 Nunca faça rebase de branches que outras pessoas estão usando. O rebase reescreve o histórico e causa divergência para quem colabora.

Squash merge — compacta em um commit

squash
$ git switch main
$ git merge --squash feature/login
$ git commit -m 'Implementa sistema de login'

Cherry-pick — copia um commit específico

cherry-pick
$ git cherry-pick <commit-id>          # copia um commit para a branch atual
$ git cherry-pick <c1>..<c2>          # copia um intervalo de commits
MétodoResultadoQuando usar
mergeCria commit de mergePreservar histórico completo
rebaseHistórico linearBranch local ainda não publicada
squash mergeUm único commitLimpar histórico antes de integrar
cherry-pickCopia commit avulsoTrazer só uma correção específica
09

Sincronizando com o Remoto — Pull e Push

Comandos para enviar e receber mudanças entre o repositório local e o remoto.

git push — enviar commits

git push
$ git push origin main              # envia branch main para origin
$ git push                          # envia para o tracking branch configurado
$ git push -u origin <branch>      # primeiro push — configura tracking
$ git push --tags                   # envia todas as tags
$ git push --force-with-lease      # force push seguro (verifica conflitos)
$ git push origin --delete <branch>  # deleta branch no remoto
⚠️ Prefira --force-with-lease ao invés de --force. O --force-with-lease falha se alguém fez push enquanto você trabalhava, evitando sobrescrever trabalho alheio.

git pull e git fetch — receber mudanças

git pull e git fetch
$ git fetch origin              # baixa mudanças SEM aplicar — seguro para inspecionar
$ git fetch origin main         # busca só a branch main
$ git pull                      # fetch + merge na branch atual
$ git pull origin main          # especificando remote e branch
$ git pull --rebase            # fetch + rebase (histórico mais limpo)
$ git pull upstream main        # busca do repositório upstream (Fork)
git fetch

Baixa mudanças sem alterar seus arquivos. Use para ver o que mudou antes de integrar.

git pull

Atalho para fetch + merge. Aplica as mudanças remotas direto na branch atual.

git pull --rebase

Atalho para fetch + rebase. Histórico linear, sem commits de merge desnecessários.

Tags — marcando versões

git tag
$ git tag                       # lista todas as tags
$ git tag v1.0.0                # cria tag leve no commit atual
$ git tag -a v1.0.0 -m "Release 1.0"  # tag anotada com mensagem
$ git push --tags               # envia todas as tags para o remoto
$ git push origin v1.0.0        # envia tag específica
10

Gerenciando Remotos — GitHub

Tudo relacionado à conexão com repositórios remotos: adicionar, listar, corrigir, remover e trabalhar com Fork.

Adicionar e listar remotos

git remote — gerenciar conexões
$ git remote -v                              # lista remotos e suas URLs
$ git remote add origin <url>               # adiciona remote com apelido origin
$ git remote add upstream <url>             # adiciona upstream (para Forks)
origin    https://github.com/usuario/repo.git (fetch)
origin    https://github.com/usuario/repo.git (push)
upstream  https://github.com/original/repo.git (fetch)
upstream  https://github.com/original/repo.git (push)

Corrigir e remover remotos

corrigir URL e remover
$ git remote set-url origin <nova-url>      # troca a URL sem remover o remote
$ git remote remove origin                  # remove completamente o remote "origin"
$ git remote rename origin upstream         # renomeia um remote
ℹ️ Quando usar cada um: use set-url para corrigir só a URL. Use remove quando quiser desconectar completamente — como ao pegar um projeto clonado e publicar como repositório próprio.

Remover um commit já publicado no remoto

Se você fez push de um commit errado e precisa removê-lo do GitHub, o processo envolve dois passos: desfazer localmente e forçar o push.

removendo commit do remoto — use com cuidado
# Opção 1 — revert (mais seguro, preserva histórico)
$ git revert <commit-id>          # cria commit que desfaz o erro
$ git push origin main            # envia normalmente

# Opção 2 — reset + force push (apaga do histórico ⚠️ avise o time)
$ git reset HEAD^                 # desfaz o commit localmente
$ git push --force-with-lease    # sobrescreve o remoto

# Opção 3 — remover arquivo específico do histórico inteiro
$ git filter-branch --force --index-filter \
  'git rm --cached --ignore-unmatch segredo.env' HEAD
$ git push --force-with-lease    # use quando commitou senha ou .env
🚨 Se você publicou credenciais (senhas, tokens, chaves), considere-as comprometidas — revogue e gere novas imediatamente, mesmo antes de remover do histórico. Bots varrem o GitHub em segundos.

Fluxo completo — publicar repositório novo

do git init ao primeiro push
$ echo "# projeto" >> README.md
$ git init
$ git add README.md
$ git commit -m "first commit"
$ git branch -M main
$ git remote add origin <url>
$ git push -u origin main

Fluxo Fork — contribuir em projetos de terceiros

fork + upstream
# 1. Fork pelo GitHub → clone do SEU Fork
$ git clone https://github.com/SEU-USUARIO/projeto.git

# 2. Configure o repositório original como upstream
$ git remote add upstream https://github.com/DONO-ORIGINAL/projeto.git

# 3. Antes de trabalhar — sempre atualize do upstream
$ git pull upstream main

# 4. Trabalhe em uma branch com seu nome
$ git switch -c feature/seu-nome/sua-parte

# 5. Envie para o SEU Fork e abra Pull Request
$ git push origin feature/seu-nome/sua-parte

# 6. Após merge aprovado — sincronize seu Fork
$ git switch main
$ git pull upstream main
$ git push origin main
ComandoO que faz
git remote -vLista remotos e URLs configurados
git remote add origin <url>Conecta ao repositório remoto
git remote add upstream <url>Conecta ao repositório original (Fork)
git remote set-url origin <url>Corrige a URL sem remover o remote
git remote remove originRemove completamente o remote
git remote rename origin upstreamRenomeia um remote

Configuração Global

Configurações que se aplicam a todos os repositórios do seu sistema.

Configurações essenciais

git config
# Identidade — obrigatório antes do primeiro commit
$ git config --global user.name 'Seu Nome'
$ git config --global user.email 'email@exemplo.com'

# Editor padrão — abre no VS Code
$ git config --global core.editor "code --wait"

# Branch inicial — padrão moderno
$ git config --global init.defaultBranch main

# Ver todas as configurações
$ git config --global --list

# Aliases úteis
$ git config --global alias.st status
$ git config --global alias.lg "log --oneline --graph --all"
$ git config --global alias.undo "reset HEAD^"

Arquivos importantes

ArquivoOnde ficaDescrição
.git/configDentro do projetoConfigurações locais do repositório atual
~/.gitconfigPasta home do usuárioConfigurações globais — todos os repositórios
.gitignoreRaiz do projetoArquivos e pastas que o Git deve ignorar
aliases recomendados — ~/.gitconfig
[alias]
    st   = status
    co   = checkout
    br   = branch
    lg   = log --oneline --graph --all
    undo = reset HEAD^
    save = stash push -m
    pub  = push -u origin HEAD
