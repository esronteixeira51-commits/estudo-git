Exercício 01 fácil

Seu primeiro repositório completo

⏱ Duração
5–10 minutos
🎯 Objetivo
Criar repositório, commitar e ver histórico
📚 Seção na apostila
Apostila 2 — Seções 7, 9
📖 Situação

Você acabou de criar um projeto do zero — três arquivos: um README, um HTML e um CSS. Quer que o Git comece a rastrear esse projeto e salvar o histórico de mudanças.

1
Inicialize o repositório Git
Na pasta treino-git, execute o comando abaixo. Uma pasta oculta .git/ será criada — isso transforma a pasta em um repositório.
2
Crie os arquivos do projeto
Use o terminal para criar os três arquivos de uma vez.
3
Veja o que o Git enxerga
O git status mostra os arquivos que o Git ainda não rastreia. Devem aparecer os 3 arquivos como "Untracked files".
4
Adicione e commite
Adicione todos ao staging e faça o primeiro commit.
5
Confirme o resultado no histórico
Use o git log para ver o commit registrado.
exercício 01 — terminal 
# Passo 1 — inicialize
$ git init
Initialized empty Git repository in /treino-git/.git/

# Passo 2 — crie os arquivos
$ echo "# Meu Projeto" >> README.md
$ touch index.html style.css

# Passo 3 — veja o estado
$ git status
Untracked files:
  index.html
  README.md
  style.css

# Passo 4 — adicione tudo e commite
$ git add .
$ git commit -m "first commit: estrutura inicial"
[main (root-commit) 3e887ab] first commit: estrutura inicial
 3 files changed, 1 insertion(+)

# Passo 5 — veja o histórico
$ git log --oneline
3e887ab (HEAD -> main) first commit: estrutura inicial
✅
Resultado esperado

O git log --oneline mostra exatamente 1 commit com a mensagem que você escreveu. Se aparecer assim, o exercício está concluído com sucesso.

Comandos usados neste exercício
git init git status git add . git commit -m git log --oneline

Exercício 02 fácil

Trabalhando com branches

⏱ Duração
10–15 minutos
🎯 Objetivo
Criar branch, commitar e fazer merge
📚 Seção na apostila
Apostila 2 — Seção 8
📖 Situação

Você precisa adicionar uma página "Sobre nós" ao projeto, mas não quer mexer no código principal enquanto ainda está desenvolvendo. A solução é criar uma branch separada, trabalhar nela e depois integrar ao main.

⚠️
Pré-requisito

Este exercício continua de onde o Exercício 1 parou. Se você ainda não fez o Exercício 1, volte e complete-o antes de continuar.

exercício 02 — terminal 
# 1. Crie e entre na nova branch
$ git switch -c feature/pagina-sobre
Switched to a new branch 'feature/pagina-sobre'

# 2. Crie um arquivo NESSA branch
$ echo "Sobre nós" >> sobre.html
$ git add sobre.html
$ git commit -m "Adiciona página sobre"

# 3. Volte para main e veja que sobre.html NÃO existe aqui
$ git switch main
$ ls
index.html  README.md  style.css   ← sobre.html não aparece!

# 4. Faça o merge da feature no main
$ git merge feature/pagina-sobre
Fast-forward
 sobre.html | 1 +

# 5. Agora sobre.html existe no main
$ ls
index.html  README.md  sobre.html  style.css  ← apareceu!

# 6. Delete a branch (trabalho concluído)
$ git branch -d feature/pagina-sobre

# 7. Veja o histórico com o grafo
$ git log --oneline --graph
✅
Resultado esperado

No passo 3, sobre.html não existe na branch main. Após o merge no passo 5, o arquivo aparece. Isso prova que as branches são isoladas — mudanças em uma não afetam a outra até o merge.

Comandos usados neste exercício
git switch -c git switch git merge git branch -d git log --graph

Exercício 03 médio

Desfazendo um erro

⏱ Duração
10–15 minutos
🎯 Objetivo
Usar reset e restore para desfazer mudanças
📚 Seção na apostila
Apostila 1 — Seção 8
📖 Situação

Você fez um commit com um erro — adicionou uma linha errada no README. Precisa voltar atrás sem perder o resto do trabalho. O Git tem dois níveis de "desfazer": remover o commit do histórico, e descartar a mudança no arquivo.

exercício 03 — terminal 
# 1. Simule um erro — adicione uma linha ruim e commite
$ echo "linha com ERRO aqui" >> README.md
$ git add README.md
$ git commit -m "commit com erro"

# 2. Veja que o commit existe no histórico
$ git log --oneline
a4b5c6d (HEAD -> main) commit com erro
3e887ab first commit: estrutura inicial

# 3. Desfaça o commit — mantém o arquivo editado
$ git reset HEAD^

# 4. O commit sumiu, mas o arquivo ainda tem a linha
$ git log --oneline
3e887ab (HEAD -> main) first commit: estrutura inicial

$ cat README.md
# Meu Projeto
linha com ERRO aqui       ← ainda está aqui

# 5. Agora descarte a mudança do arquivo também
$ git restore README.md

# 6. Confirme que a linha errada sumiu do arquivo
$ cat README.md
# Meu Projeto
✅
Resultado esperado

Após o git reset HEAD^ o commit some do log, mas o arquivo ainda tem a linha errada. Após o git restore o arquivo volta ao estado original. Você aprendeu a diferença entre desfazer um commit e desfazer uma mudança no arquivo.

💡
Por que dois comandos?

git reset HEAD^ move o ponteiro do histórico — o commit some, mas o arquivo fica editado (útil se você queria reescrever o commit). git restore descarta a edição do arquivo — volta ao último commit registrado. São níveis diferentes de "desfazer".

Comandos usados neste exercício
git reset HEAD^ git restore cat git log --oneline

Exercício 04 médio

Unificando main e master

⏱ Duração
15–20 minutos
🎯 Objetivo
Merge com históricos não relacionados
📚 Seção na apostila
Apostila 2 — Seção 14
📖 Situação

Você criou um repositório localmente com git init (que gerou a branch master). O GitHub criou a branch main separadamente com um README automático. Agora os dois têm históricos completamente independentes — o Git recusa o merge. Você precisa unificá-los.

🚨
Atenção

Este exercício usa uma pasta nova separada para não misturar com os exercícios anteriores. Siga o passo 1 com atenção.

exercício 04 — terminal 
# 1. Crie uma pasta nova para este exercício
$ cd ..
$ mkdir ex04-merge && cd ex04-merge
$ git init

# 2. Crie o branch "master" com um commit (simula repositório local)
$ echo "arquivo do repositório local" >> local.txt
$ git add local.txt
$ git commit -m "commit do repositório local"

# 3. Crie o branch "main" com histórico separado (simula o GitHub)
$ git checkout --orphan main
$ git rm -rf .
$ echo "arquivo do GitHub" >> github.txt
$ git add github.txt
$ git commit -m "commit do GitHub — histórico separado"

# 4. Veja que os dois branches não compartilham histórico
$ git log --oneline --all
b1c2d3e (HEAD -> main) commit do GitHub — histórico separado
a4b5c6d (master) commit do repositório local

# 5. Vá para master e tente o merge normal — vai FALHAR
$ git switch master
$ git merge main
fatal: refusing to merge unrelated histories

# 6. Use a flag correta para resolver
$ git merge main --allow-unrelated-histories
Merge made by the 'ort' strategy.
 github.txt | 1 +
 1 file changed, 1 insertion(+)

# 7. Confirme que os dois arquivos agora coexistem
$ ls
github.txt  local.txt

# 8. Veja o histórico unificado com o grafo
$ git log --oneline --graph --all
✅
Resultado esperado

O passo 5 mostra o erro refusing to merge unrelated histories. O passo 6 resolve com a flag. Ao final, local.txt e github.txt existem juntos e o --graph mostra duas linhas convergindo em um ponto de merge.

Comandos usados neste exercício
git checkout --orphan git branch -a git merge --allow-unrelated-histories git log --graph --all

Exercício 05 avançado

Publicando no GitHub do zero

⏱ Duração
15–20 minutos
🎯 Objetivo
Fluxo completo do git init ao git push
📚 Seção na apostila
Apostila 2 — Seções 10, 12
📖 Situação

Você tem um projeto local pronto e quer publicar no GitHub pela primeira vez. Esse é o fluxo exato que o próprio GitHub mostra quando você cria um repositório novo e já tem código local.

⚠️
Pré-requisito

Você precisa de uma conta no GitHub. Acesse github.com/new e crie um repositório vazio chamado meu-primeiro-repo. Não marque "Initialize with README" — o repositório deve ficar completamente vazio.

exercício 05 — terminal 
# 1. Crie e entre na pasta do projeto
$ mkdir meu-primeiro-repo && cd meu-primeiro-repo

# 2. Crie o README e inicialize
$ echo "# Meu Primeiro Repositório" >> README.md
$ git init
$ git add README.md
$ git commit -m "first commit"

# 3. Garanta que o branch se chama main
$ git branch -M main

# 4. Conecte ao GitHub (substitua pela sua URL)
$ git remote add origin https://github.com/SEU-USUARIO/meu-primeiro-repo.git

# 5. Confirme a URL antes de enviar
$ git remote -v
origin  https://github.com/SEU-USUARIO/meu-primeiro-repo.git (fetch)
origin  https://github.com/SEU-USUARIO/meu-primeiro-repo.git (push)

# 6. Envie para o GitHub (primeiro push com -u)
$ git push -u origin main
Enumerating objects: 3, done.
Writing objects: 100% (3/3), done.
Branch 'main' set up to track remote branch 'main' from 'origin'.

# 7. Abra o navegador e confirme que o README aparece no repositório ✅
✅
Resultado esperado

Ao abrir o GitHub no navegador, o repositório mostra o arquivo README.md renderizado com o título "Meu Primeiro Repositório". A partir de agora, para enviar novos commits basta usar git push — sem precisar do -u origin main novamente.

Comandos usados neste exercício
git branch -M git remote add origin git remote -v git push -u origin main

Exercício 06 avançado

Corrigindo um remote errado

⏱ Duração
10 minutos
🎯 Objetivo
Remover e reconfigurar o origin
📚 Seção na apostila
Apostila 2 — Seção 13
📖 Situação

Você digitou a URL errada no git remote add origin e só percebeu depois. O push vai falhar ou ir para o lugar errado. Você precisa remover a conexão errada e reconfigurar com a URL correta — tudo sem perder nenhum commit local.

exercício 06 — terminal 
# 1. Simule o erro — adicione uma URL errada
$ git remote add origin https://github.com/usuario/repositorio-errado.git

# 2. Veja a URL errada configurada
$ git remote -v
origin  https://github.com/usuario/repositorio-errado.git (fetch)
origin  https://github.com/usuario/repositorio-errado.git (push)

# 3. Remova o remote errado
$ git remote remove origin

# 4. Confirme que foi removido (sem saída = sucesso)
$ git remote -v

# 5. Adicione a URL correta
$ git remote add origin https://github.com/SEU-USUARIO/repositorio-correto.git

# 6. Confirme a URL correta
$ git remote -v
origin  https://github.com/SEU-USUARIO/repositorio-correto.git (fetch)
origin  https://github.com/SEU-USUARIO/repositorio-correto.git (push)

# ─── ALTERNATIVA MAIS RÁPIDA ──────────────────────────────
# Se só quiser TROCAR a URL sem remover e readicionar:
$ git remote set-url origin https://github.com/SEU-USUARIO/repositorio-correto.git
✅
Resultado esperado

Após o git remote -v no passo 4 não aparece nenhuma saída — significa que não há remotes configurados. Após o passo 6, a URL correta aparece. Nenhum arquivo local foi alterado durante todo o processo.

💡
Quando usar cada abordagem

Use git remote remove + git remote add quando quiser desconectar completamente e reconfigurar do zero. Use git remote set-url quando só precisar trocar a URL — é mais direto e rápido.

Comandos usados neste exercício
git remote -v git remote remove origin git remote add origin git remote set-url origin

Exercício 07 fácil

PC formatado — recuperando o projeto

⏱ Duração
5–10 minutos
🎯 Objetivo
Recuperar projeto do GitHub após formatar
📚 Seção na apostila
Apostila 2 — Seções 2, 10
📖 Situação

Você tinha um projeto no PC, com repositório no GitHub. Formatou o PC e perdeu os arquivos locais. O projeto continua intacto no GitHub — você precisa recuperar tudo e voltar a trabalhar como se nada tivesse acontecido.

💡
O que realmente aconteceu

Você não perdeu o projeto — perdeu apenas a cópia local. O GitHub tem uma cópia completa de todos os arquivos, commits e branches. O Git é distribuído — o repositório oficial vive no GitHub, não no seu PC.

1
Instale o Git e configure seus dados
Após formatar, reinstale o Git e configure nome e e-mail — use os mesmos dados da conta GitHub.
2
Clone o repositório do GitHub
Um único comando traz tudo de volta: arquivos, histórico, branches e configuração do remote.
3
Abra no VS Code e continue trabalhando
Como se nunca tivesse formatado.
exercício 07 — terminal 
# Passo 1 — configure o Git na máquina nova/formatada
$ git config --global user.name "Seu Nome"
$ git config --global user.email "seu@email.com"

# Passo 2 — clone o repositório do GitHub
# Traz: arquivos + histórico + branches + remote configurado
$ git clone https://github.com/seu-usuario/seu-projeto.git
Cloning into 'seu-projeto'...
remote: Enumerating objects: 143, done.
Receiving objects: 100% (143/143), done.

# Passo 3 — entre na pasta e abra no VS Code
$ cd seu-projeto
$ code .

# Confirme que o remote está configurado
$ git remote -v
origin  https://github.com/seu-usuario/seu-projeto.git (fetch)
origin  https://github.com/seu-usuario/seu-projeto.git (push)

# Veja o histórico completo — tudo foi recuperado
$ git log --oneline
f1c2d3e (HEAD -> main) último commit antes de formatar
a4b5c6d commit anterior
3e887ab first commit
✅
Resultado esperado

Todos os arquivos estão de volta. O git log mostra o histórico completo. O remote origin já está configurado — você pode continuar com git push normalmente, sem precisar reconfigurar nada.

⚠️
Boa prática que evita esse drama

Sempre faça git push antes de qualquer formatação ou troca de máquina. O que não está no GitHub não existe para o Git. Trate o GitHub como seu único backup confiável.

Comandos usados neste exercício
git config --global git clone git remote -v git log --oneline

Exercício 08 fácil

Dois PCs, um projeto — trocando de máquina

⏱ Duração
10 minutos
🎯 Objetivo
Sincronizar trabalho entre duas máquinas
📚 Seção na apostila
Apostila 2 — Seção 10
📖 Situação

Você trabalha no PC em casa e às vezes no notebook quando está viajando ou em outro lugar. Quer continuar o mesmo projeto nas duas máquinas sem perder nada e sem criar conflitos.

💡
O GitHub é a ponte entre as máquinas

Nenhuma máquina é a "oficial" — o GitHub é. As máquinas só são onde você trabalha. A regra é simples: sempre comece com git pull e sempre termine com git push.

exercício 08 — fluxo entre duas máquinas 
# ── PC DE CASA — antes de sair ────────────────────────────
$ git add .
$ git commit -m "parando aqui — continuando no notebook"
$ git push origin main
To github.com/usuario/projeto.git
   f1c2d3e..a4b5c6d  main -> main

# ── NOTEBOOK — primeira vez (clone uma única vez) ─────────
$ git clone https://github.com/usuario/projeto.git
$ cd projeto

# ── NOTEBOOK — todas as vezes seguintes ───────────────────
$ git pull
Updating f1c2d3e..a4b5c6d
Fast-forward
 index.html | 5 +++--

# ... trabalha no notebook ...
$ git add .
$ git commit -m "continuação no notebook"
$ git push origin main

# ── PC DE CASA — ao voltar ────────────────────────────────
$ git pull
Updating a4b5c6d..7d8e9f0
Fast-forward
 style.css | 12 ++++++++++++

# Continua de onde o notebook parou ✅
✅
Resultado esperado

O trabalho flui entre as duas máquinas sem conflito. O clone é feito apenas uma vez por máquina — depois disso é só git pull para sincronizar. O GitHub funciona como ponto central entre todas as máquinas.

⚠️
O que causa conflito nesse cenário

Editar o mesmo arquivo nas duas máquinas sem dar git pull antes. A regra de ouro: sempre pull antes de começar, sempre push antes de parar.

Comandos usados neste exercício
git pull git push origin main git clone git add . git commit -m

Exercício 09 médio

Equipe — duas pessoas, arquivos diferentes

⏱ Duração
15–20 minutos
🎯 Objetivo
Colaboração sem conflito em arquivos diferentes
📚 Seção na apostila
Apostila 2 — Seções 8, 10
📖 Situação

Você e um colega estão trabalhando no mesmo projeto. Cada um ficou responsável por uma parte diferente — você cuida do login.js e seu colega do dashboard.js. Os dois querem commitar e fazer push ao mesmo tempo.

💡
Por que funciona sem conflito

O Git rastreia mudanças por arquivo. Como cada pessoa tocou em arquivos diferentes, o GitHub aceita os dois pushes sem problema — ele sabe exatamente o que cada um fez e onde.

exercício 09 — simule os dois alunos no mesmo terminal 
# ── SETUP — crie o projeto base ───────────────────────────
$ mkdir projeto-equipe && cd projeto-equipe
$ git init
$ touch login.js dashboard.js
$ git add .
$ git commit -m "estrutura inicial"

# ── ALUNO A — trabalha em login.js ────────────────────────
$ git switch -c feature/aluno-a/login
$ echo "function validarLogin() {}" >> login.js
$ git add login.js
$ git commit -m "Aluno A: adiciona validação de login"

# ── ALUNO B — trabalha em dashboard.js ───────────────────
$ git switch main
$ git switch -c feature/aluno-b/dashboard
$ echo "function carregarDados() {}" >> dashboard.js
$ git add dashboard.js
$ git commit -m "Aluno B: adiciona carregamento de dados"

# ── JUNTANDO OS DOIS NA MAIN ──────────────────────────────
$ git switch main
$ git merge feature/aluno-a/login
Fast-forward — login.js | 1 +

$ git merge feature/aluno-b/dashboard
Fast-forward — dashboard.js | 1 +

# Confirme que os dois arquivos têm o conteúdo de cada aluno
$ cat login.js
function validarLogin() {}

$ cat dashboard.js
function carregarDados() {}

# Histórico mostrando as duas contribuições
$ git log --oneline
a4b5c6d Aluno B: adiciona carregamento de dados
f1c2d3e Aluno A: adiciona validação de login
3e887ab estrutura inicial
✅
Resultado esperado

Os dois merges acontecem sem conflito. Cada arquivo tem o conteúdo do seu respectivo aluno. O histórico registra claramente quem fez o quê. Esse é o fluxo ideal de trabalho em equipe.

Comandos usados neste exercício
git switch -c git switch main git merge git log --oneline cat

Exercício 10 médio

Equipe — mesmo arquivo, resolvendo conflito

⏱ Duração
20–25 minutos
🎯 Objetivo
Identificar e resolver um conflito de merge
📚 Seção na apostila
Apostila 2 — Seção 8
📖 Situação

Dois alunos precisaram trabalhar no mesmo arquivo calculos.js. O Aluno A definiu o desconto como 15% e o Aluno B definiu como 20%. Agora querem juntar o trabalho — mas o Git não sabe qual versão é a correta e precisa da sua decisão.

🚨
O conflito não é um erro — é uma pergunta

Quando o Git abre um conflito, ele está dizendo: "duas pessoas mudaram a mesma linha — eu não sei qual é a certa, você decide." Não há nada quebrado — é só uma decisão que precisa ser tomada por um humano.

exercício 10 — criando e resolvendo um conflito 
# ── SETUP ─────────────────────────────────────────────────
$ mkdir conflito-equipe && cd conflito-equipe
$ git init
$ echo "function calcularDesconto(valor) { return valor * 0.10; }" >> calculos.js
$ git add . && git commit -m "estrutura inicial"

# ── ALUNO A — muda para 15% ───────────────────────────────
$ git switch -c feature/aluno-a
$ echo "function calcularDesconto(valor) { return valor * 0.15; }" > calculos.js
$ git add calculos.js
$ git commit -m "Aluno A: desconto de 15%"

# ── ALUNO B — muda para 20% ───────────────────────────────
$ git switch main
$ git switch -c feature/aluno-b
$ echo "function calcularDesconto(valor) { return valor * 0.20; }" > calculos.js
$ git add calculos.js
$ git commit -m "Aluno B: desconto de 20%"

# ── JUNTANDO — o conflito aparece ────────────────────────
$ git switch main
$ git merge feature/aluno-a
Fast-forward — ok, Aluno A entrou sem problema

$ git merge feature/aluno-b
CONFLICT (content): Merge conflict in calculos.js
Automatic merge failed — fix conflicts and then commit.

# ── O ARQUIVO AGORA ESTÁ ASSIM ────────────────────────────
$ cat calculos.js
<<<<<<< HEAD (Aluno A — já está na main)
function calcularDesconto(valor) { return valor * 0.15; }
=======
function calcularDesconto(valor) { return valor * 0.20; }
>>>>>>> feature/aluno-b

# ── DECISÃO — os dois conversam e decidem juntos ──────────
# Opção 1: fica com o 15% (Aluno A vence)
# Opção 2: fica com o 20% (Aluno B vence)
# Opção 3: cria uma versão que combina os dois (melhor)

# Editam o arquivo manualmente — versão combinada:
$ echo "function calcularDesconto(valor, tipo) { return tipo === 'vip' ? valor * 0.20 : valor * 0.15; }" > calculos.js

# ── FINALIZANDO O MERGE ───────────────────────────────────
$ git add calculos.js
$ git commit -m "Resolve conflito: combina desconto vip (20%) e padrão (15%)"

# Confirme o resultado final
$ cat calculos.js
function calcularDesconto(valor, tipo) { return tipo === 'vip' ? valor * 0.20 : valor * 0.15; }

$ git log --oneline
b2c3d4e Resolve conflito: combina desconto vip (20%) e padrão (15%)
a4b5c6d Aluno A: desconto de 15%
3e887ab estrutura inicial
✅
Resultado esperado

O conflito aparece, você identifica as duas versões entre os marcadores <<<<<<< e >>>>>>>, edita o arquivo com a versão correta, remove os marcadores, faz o commit. O projeto continua com o melhor dos dois trabalhos.

💡
Como evitar esse conflito da próxima vez

Antes de dois alunos tocarem no mesmo arquivo, dividam responsabilidades dentro do arquivo — Aluno A cuida da função X, Aluno B cuida da função Y. Conflito no Git é reflexo de falta de comunicação na equipe.

Comandos usados neste exercício
git merge git status git add (após resolver) git commit -m cat

Exercício 11 avançado

Fork, upstream e Pull Request — fluxo de turma

⏱ Duração
25–30 minutos
🎯 Objetivo
Contribuir em um projeto via Fork e Pull Request
📚 Seção na apostila
Apostila 2 — Seções 10, 12
📖 Situação

O professor criou um repositório oficial da turma no GitHub. Cada aluno vai contribuir com sua parte do projeto — mas ninguém tem acesso direto ao repositório oficial. O fluxo correto é: fazer um Fork, trabalhar no Fork próprio e propor a contribuição via Pull Request para o professor revisar e aprovar.

💡
Por que Fork e não acesso direto

Com acesso direto, qualquer aluno pode quebrar o projeto principal. Com Fork + Pull Request, o professor revisa cada contribuição antes de entrar no projeto. É o mesmo modelo usado em projetos open source como o VS Code e o React.

⚠️
Pré-requisito

Para este exercício você precisa de uma conta no GitHub e de um repositório de um colega (ou professor) para fazer o Fork. Peça a URL do repositório antes de começar.

1
Professor — cria o repositório oficial
Cria o repositório no GitHub com a estrutura inicial do projeto. Não dá acesso de escrita para os alunos.
2
Aluno — faz o Fork
Acessa a página do repositório do professor no GitHub e clica no botão Fork. Isso cria uma cópia do projeto na conta do aluno.
3
Aluno — clona o Fork e configura upstream
Clona o seu Fork (não o do professor) e adiciona o repositório do professor como upstream para receber atualizações futuras.
4
Aluno — trabalha e envia para o próprio Fork
Cria uma branch, faz as mudanças, commita e faz push para o Fork dele — não para o repositório do professor.
5
Aluno — abre um Pull Request
No GitHub, abre um Pull Request do Fork para o repositório do professor. Descreve o que foi feito.
6
Professor — revisa e aprova
Analisa o código, faz comentários se necessário e faz o merge quando estiver aprovado.
exercício 11 — terminal do aluno 
# Passo 1 — clone o SEU Fork (não o do professor)
$ git clone https://github.com/SEU-USUARIO/projeto-turma.git
$ cd projeto-turma

# Passo 2 — adicione o repositório do professor como upstream
$ git remote add upstream https://github.com/PROFESSOR/projeto-turma.git

# Passo 3 — confirme os dois remotes configurados
$ git remote -v
origin    https://github.com/SEU-USUARIO/projeto-turma.git (fetch)
origin    https://github.com/SEU-USUARIO/projeto-turma.git (push)
upstream  https://github.com/PROFESSOR/projeto-turma.git (fetch)
upstream  https://github.com/PROFESSOR/projeto-turma.git (push)

# Passo 4 — sempre atualize do professor antes de começar
$ git pull upstream main

# Passo 5 — crie uma branch com seu nome e sua parte
$ git switch -c feature/seu-nome/sua-parte

# Passo 6 — trabalhe e commite
$ echo "minha contribuição" >> minha-parte.js
$ git add .
$ git commit -m "Adiciona módulo de cadastro de usuários"

# Passo 7 — envie para o SEU Fork
$ git push origin feature/seu-nome/sua-parte
To https://github.com/SEU-USUARIO/projeto-turma.git
 * [new branch] feature/seu-nome/sua-parte -> feature/seu-nome/sua-parte

# Passo 8 — abra o Pull Request no GitHub
# Acesse github.com/SEU-USUARIO/projeto-turma
# Clique em "Compare & pull request"
# Escreva uma descrição clara do que foi feito
# Clique em "Create pull request"

# ── APÓS O PROFESSOR FAZER O MERGE ────────────────────────
# Atualize seu Fork com o merge aprovado
$ git switch main
$ git pull upstream main
$ git push origin main

# Delete a branch — contribuição concluída
$ git branch -d feature/seu-nome/sua-parte
✅
Resultado esperado

Seu Fork tem dois remotes: origin (seu Fork) e upstream (repositório do professor). Você trabalha no origin e só recebe do upstream. O Pull Request aparece na página do repositório do professor para revisão. Após o merge, sua contribuição faz parte do projeto oficial.

💡
Regras de ouro para contribuição em turma

→ Nunca trabalhe direto na main — sempre em uma branch com seu nome
→ Sempre dê git pull upstream main antes de começar
→ Um Pull Request por funcionalidade — não misture tudo em um só
→ Descreva o Pull Request com clareza — o professor precisa entender o que foi feito
→ Atualize seu Fork após cada merge aprovado

Comandos usados neste exercício
git remote add upstream git pull upstream main git switch -c git push origin branch git branch -d
